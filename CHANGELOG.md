# Changelog

## v0.2.1 — 2026-07-14

**质量闸：自检清单从"人肉过"变成"机器判"**

起因（鲁班工坊活体检查）：拿三份真实报告对着 v0.2.0 的自检清单机检，**两份 FAIL**——城投攒了 11 个孤儿词条、旅行社图号重号（[1,2,2,3,3,4,5]）、两份图里都明晃晃写着"要素/输入/输出"。**这些全都是清单里白纸黑字的项。** 结论不是"报告没跟上 skill"，而是：**人肉过的清单必然被跳过；唯一被脚本化的那条规则（`check_svg_overlap.py`）是唯一真正在跑的规则。**

- 新增 [`scripts/check_report.py`](scripts/check_report.py)（零依赖，纯标准库）：机检 7 类硬规则——章节骨架（2.3/2.4 分立、3.3 方案建议）、图号连续、图注完整、术语死链、术语孤儿、图内抽象词、SVG 硬编码色；另报 3 类 WARN（正文抽象词、图解元话语、系统维度用语）供人工判断。
- **接进工作流**：第 7 步「机器自检」两个脚本并列，**FAIL 必须清零**；第 9 步「交付」升级为**质量闸——不过闸不许导 PDF / 同步飞书**（学 astra-research-open 的"不过闸就不生成 PDF"）。
- **自检清单重构**：机检项标 🤖 并移交脚本，人肉项只留机器判不了的（素材忠实度、生动程度、数字溯源、20 秒能否读懂）——同一条规则不再既靠人肉又靠机器。
- 回放验证：三份真实报告，抓出 12 项 FAIL，**央国企（唯一按 v0.2.0 写的）全绿 0 FAIL 0 WARN——零误报**。
- **回归纪律**（立成明文规矩）：规则一旦变更，必须拿手头所有历史报告重跑 `check_report.py`——规则改了而旧报告没跟上，要么该返修报告，要么该反省这条规则是不是拍脑袋定的。

**README 重写（定位 + 首屏信任）**
- 首屏钩子换成真正的差异点：**"别人拿全网二手信息写给行家看，它拿你的一手访谈写给小白看"**——生态搜索显示"访谈转写 → 行业研报"这条路 GitHub 上无人认领，而现有同类清一色是公开网搜 + PE 腔，默认读者已懂行。原 README 在介绍五段框架，白瞎了真正的护城河。
- Showcase 用**质量闸的真实输出**（从城投/旅行社真报告里抓出的 12 条 FAIL），不用编造样例：别的 skill 只能在 prompt 里写"请注意数字准确"，我们能甩出"术语孤儿 11 个、图号重号、图里写着'要素'"。零敏感，可直接传播。
- 新增差异化对比表（信息源 / 读者 / 抽象词 / 质量保证 / 产物）、安全边界、触发方式、`npx skills add` 一行安装、skills.sh 徽章、LICENSE(MIT)。
- **明确不收录样例报告**：真实报告含客户访谈里的业务细节（灰色环节、利润结构），属敏感内容，不进公开仓库。

## v0.2.0 — 2026-07-14

破坏性改版：基于央国企电商报告的整轮打磨（2.2 抽象、2.3 图看不懂、利润讲不清、方案切入太虚）沉淀。按旧版写出的 2.2/2.3/3.3 会被直接返工，故升次版本号。

**章节骨架（破坏性）**
- `2.3 上下游与利益分配` **拆成两节**：`2.3 上下游业务协作链`（只讲谁和谁协作、交接什么，不出现金额）+ `2.4 资金流转与利益分配`（只讲钱与票，不重复单与货）。上下游角色表跟着协作链走，落在 2.3。
- 骨架图 **4 类 → 5 类**：新增「业务协作链图」与「资金流转图」分立（原"上下游链条全景图"一张图同时塞协作和资金，读者看不懂）。
- `3.3 方案切入视角` → **`3.3 方案建议`**：痛点一两句话收掉（不列大表），改为按 **P0–P3** 列价值点，每条落到"哪个角色期望实现什么 + 对应痛点"，收尾必须给一条边界（哪里别碰）。

**图的画法**
- **协作链图**新规范：按上下游维度**单向排布**、**从业务起点出发**（起点是客户就客户在左、上游在右，不迁就"上游习惯画左边"）、箭头用 ⓪①②… 标出一单生意走完的顺序、实线=合规链 / 金虚线=灵活链、底注只写一句题眼。删掉旧的"竖版/横版双层选型"（竖版会丢掉顺序这条主线）。
- **利润账并入资金流转图**：跟一笔具体金额（如 1 万元订单）走完全程，图内一角当场算清"毛差 → 四层扣减 → 真实利润率"。取消独立的利润漏斗图。
- **图 2 删掉中间那层**：产品全景由三层（构成/生命周期/形态）改为**两层**——客户掏差价买的是哪几件事 / 装进哪几种订单形态里卖并对比。原"生命周期"层会让 2.2 读起来像第三章的业务流。

**新增铁律**
- 图内标签必须是**具体业务动作**，抽象词黑名单（要素/交付/输入/输出/职责/赋能/协同/资源整合）。
- **不许为图面整洁合并节点**：同层主体有实质差异（能开票报价高 vs 没票报价低）必须拆开画，差异就是题眼。
- 图里出现的黑话，正文或术语表必须能查到（`.term` + 附录词条）。
- 图与正文的关系：图前一句话总起、图后按图中编号分条讲解；**禁元话语**（"拿一单生意把图 3 走一遍""如图所示"）。
- 章头 lede 固定「本章回答：……？」且必须逐一对应本章小节，写完回头对一遍。

**整体梳理：去矛盾、去重复（收口）**
- 确立分工原则：**通则只写在 SKILL.md，framework.md 只写"这一节特有的东西"**，避免同一条规则在两处各写一遍、日后改一处漏一处。
- 「零素材怎么办」原本写了三遍（第 0 步 / 工作流程第 3 步 / desk-research.md）→ 收敛为：第 0 步给红线一句话，细节全在 desk-research.md。
- 「必须联网查、查到什么程度停」原本 SKILL 第 3 步与 framework 1.2/1.3 各写一遍完整规则 → framework 只留两节**特有**的取数规则（年份钉公开事实、取最新年度、个案不混进 stat 卡、大盘衬托主角）。
- 铁律「术语两级处理」+「图里的黑话必须能查到」是同一件事 → 合并为一条（铁律 9 → 8 条，无约束丢失）。
- 删掉 framework「语气与文风」里对铁律「拒绝抽象」的复述（它自己都标着"详见 SKILL.md 铁律"）、2.2 末尾重复的「图的多少由认知复杂度决定」。
- 自检清单：数字溯源两条合一、术语两条合一、交付物两条合一（20 → 18 项），去掉"不再输出 md"这类历史包袱。

**输入形态：兼容零素材（新增模式 B）**
- 新增**第 0 步分流**：先看输入是什么，再决定走哪条路。访谈素材 → 模式 A（原流程）；一句话/一个行业名/零散笔记 → **模式 B 公开资料驱动**（[references/desk-research.md](references/desk-research.md)）；**一家公司名** → 先联网查清主营与所处环节，把候选的行业与主角视角摆给用户选，确认后再开写（公司 ≠ 行业，本 skill 的主角是行业里的一类角色，不是某家公司）。
- **模式 B 的红线**：没有访谈时模型的默认行为是"用行业常理把空缺填满"，编出来的东西读着专业、不会报错——这是最危险的失败模式。故规定：能写的正常写（定义/政策/发展阶段/市场规模/产业链格局/公开玩家/模式对比），**只有访谈才拿得到的（真实利润账、灰色链路、行业黑话、一线痛点）宁可留白也不推演**；留白用 `warnbox` 显性标出，不许写成通顺的空话；骨架 5 图在模式 B 下**不是必备**，画不出真东西就不画。
- 模式 B **额外产出访谈提纲**：把每处留白反推成"该去问什么、该问谁"，按"填上之后报告会活过来"排序——这是模式 B 真正的交付价值，它把模式 B 变成模式 A。
- description 同步覆盖新触发语（"分析一下 XX 行业""研究下 XX 这家公司所在的行业""帮我摸个底"），否则零素材场景根本唤不起本 skill。

**联网检索（放开 + 定界）**
- 检索的触发条件从"1.2/1.3 两节"改为**"这句话凭什么可信"**：全文任何位置，凡属①可核实的客观事实（数字/年份/政策/行业通行规则/公开玩家）、②素材没覆盖但结构需要的内容、③受访者的全行业断言（需交叉验证，验证不了降级成"受访者认为"），都必须先查再写。
- **查到什么程度就停**：每条事实查到能标年份与来源即停；关键数字两个独立来源对上就够；查不到直接写"无公开统计"。受访者的个案经验、公司内部做法、灰色环节不必查也查不到，如实写、标口径即可。
- **新增"素材稀薄/无素材"场景**：先跟用户挑明深度会明显弱一档（内部机制写不出来），问是补素材还是接受；接受则公开资料写得到的正常写，**只有访谈才能得到的内部机制宁可留白也不推演**，页脚写明以公开资料为主。

**图与文字的分工（校准）**
- **补图不再绑定在商业模式一章**：骨架 5 图之外，全文任何位置都可加图，加不加由**内容形态**决定——关系/结构/流动/顺序/对比/配比用图，定义/判断/因果/故事/人味用文字。
- **文字不是图的附属品**：删掉"文字只承担两件事（重点结论 + 必要的场景化描述）"这种过绝的写法。报告的目标是把陌生行业**生动形象地讲给小白听**，文字本身就得生动、具体、场景化——具体的人、具体的动作、受访者原话、行业黑话、反常识处，都靠文字承载，不能写成图的干巴巴注脚。

**交付**
- **交付前必做「重建行业术语」**（写进工作流程第 9 步，不再只是一条自检项）：报告改过多轮后术语必然走样——正文新写的黑话没进附录、删掉的段落留下孤儿词条、改过的锚点变成死链。导 PDF / 同步飞书之前必须把术语过一遍，**不能省、也不能等用户提醒**；改完内容重新交付时同样要再过一遍。
- **PDF 与飞书文档改为平级的两种可选交付形态**：都放在最后一步、都要用户明确同意才做，**问清楚"要 PDF、要飞书、还是都要"由用户自己选**，不默认替他决定（原来 PDF 被写成唯一的"最终交付"，飞书只是附带）。
- 新增可选交付通道：[references/lark-sync.md](references/lark-sync.md) —— 同步到飞书文档（HTML→飞书 XML、SVG→2x PNG、占位符搬运），含 `media-insert` 进度日志混入 stdout、`block_move_after` 假成功等实战坑。
- 模板内置 `@media print { break-inside: avoid }`（图与图注、环节卡片、表格不再被 PDF 分页切开）。

## v0.1.3 — 2026-07-14

**商业模式（重点强化）**
- 明确第二章为全文重点，2.2/2.3 不能只用一张概览图带过；能用图讲清就不用大段文字。
- 2.2：行业存在两种以上实质不同的产品/订单/经营模式时，应做对比（优先图形化）；**对比维度不套固定清单**，从素材选 3–6 个最能解释"为何不能用同一套经营逻辑"的变量，图后必须给经营结论——防止千篇一律的固定维度对比表。
- 2.3：目标定为"通过产业链与资金流动讲清怎么挣钱"，**结构不定死**，但必须回答"谁付钱、收入成本在哪、利润来自哪"；交易复杂（多收入源/多笔成本/事后报账/凭证核销/需对账）时增加资金结算闭环图，简单则不硬拆。
- **图数量由认知复杂度决定**（改掉"自由发挥、不设上限"的措辞）：一图一论点、讲不清才拆；详细不等于编，边界是素材撑得住 + 可核实常识。

**图号**
- 四类骨架图改为"固定在章节位置、图号不写死为 1–4、全文按出现顺序连续编号"，修正与"商业模式按需加图"的冲突；自检项同步。

**稳健性（校准）**
- 公开数据从"必须"放软为"有则必用、查不到就明说不硬凑"，新增"大盘衬托主角、不喧宾夺主"约束。
- PDF 导出新增分页检查：长图/宽表跨页断裂的处理（`break-inside: avoid`）。
- 自检新增"过拟合回归"项：换一个结构不同的行业想一遍，规则是否会逼出僵硬套路、是否只对本案例成立。
