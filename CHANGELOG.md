## v0.2.0 — 2026-07-14

破坏性改版：基于城投定融、央国企电商、旅行社地接三份实战报告的整轮打磨沉淀（2.2 抽象、2.3 图看不懂、利润讲不清、方案切入太虚）。按 v0.1.x 写出的 2.2 / 2.3 / 3.3 会被直接返工，故升次版本号。

### 章节骨架（破坏性）
- `2.3 上下游与利益分配` **拆成两节**：`2.3 上下游业务协作链`（只讲谁和谁协作、交接什么，不出现金额）+ `2.4 资金流转与利益分配`（只讲钱与票，不重复单与货）。上下游角色表跟着协作链走，落在 2.3。
- 骨架图 **4 类 → 5 类**：「业务协作链图」与「资金流转图」分立（原一张图同时塞协作和资金，读者看不懂）。
- `3.3 方案切入视角` → **`3.3 方案建议`**：痛点一两句话收掉（不列大表），改为按 **P0–P3** 列价值点，每条落到"哪个角色期望实现什么 + 对应痛点"，收尾给一条边界（哪里别碰）。

### 图的画法
- **协作链图**：按上下游维度**单向排布、从业务起点出发**（起点是客户就客户在左，不迁就"上游习惯画左边"）、箭头 ⓪①②… 标一单生意走完的顺序、实线=合规链 / 金虚线=灵活链、底注只写一句题眼。删掉旧的"竖版/横版双层选型"。
- **资金流转图横版**：利润账**画在图里**，跟一笔具体金额（如 1 万元订单）走完，图内一角当场算"毛差 → 层层扣减 → 真实利润率"。取消独立的利润漏斗图。
- **2.4 正文按上下游流向讲，不用时间线**：和图同方向、每环一条（这一环收/付多少、赚的是什么），加粗引导词直接用角色名、不顶"下游·""上游·"前缀（图和角色表已标位置）。时间顺序是第三章 L1 的活，用时间线会和图打架、和第三章重复。
- **图 2 只两层、写法不固化**：删掉中间"生命周期"层（会让 2.2 读起来像业务流）。目的固定为 **细分 + 对比**，按行业分两类——服务型（提供哪几项服务 + 服务对比）、实体产品型（产品分哪几类、由什么构成 + 产品对比）；别把"客户图什么好处"当第一层（那是 2.1 的地盘）。

### 图与文字的分工
- **补图不绑在某一章**：骨架 5 图之外，全文任何位置都可加图，加不加由**内容形态**决定——关系/结构/流动/顺序/对比/配比用图，定义/判断/因果/故事/人味用文字。
- **文字不是图的附属品**：删掉"文字只承担重点结论 + 场景化描述"这种过绝写法。图后文字负责把图里的概念**讲活**（"点A发B""过桥"是什么），不是复述图的结构，更不写"如图所示""把图 3 走一遍"这类元话语。

### 质量闸：自检从"人肉过"变成"机器判"
- 新增 [`scripts/check_report.py`](scripts/check_report.py)（零依赖）：机检 7 类硬规则——章节骨架、图号连续、图注完整、术语死链、术语孤儿、图内抽象词、SVG 硬编码色；另报 3 类 WARN（正文抽象词、图解元话语、系统维度用语）。
- `scripts/check_svg_overlap.py` 从只查"线穿文字/文字出画布"，补上 **文字压文字**、**文字压实心块**（深底压深字读不清）两类 FAIL——都是翻修时"脚本报绿、肉眼却看出糊"逼出来的。
- **接进工作流**：第 7 步机器自检两脚本并列、FAIL 必须清零；第 9 步交付升级为**质量闸——不过闸不许导 PDF / 同步飞书**。自检清单机检项标 🤖 移交脚本，人肉项只留机器判不了的（素材忠实度、生动程度、数字溯源）。
- **回归纪律**（立成明文）：规则一旦变更，必须拿手头所有历史报告重跑脚本——旧报告没跟上，要么返修报告、要么反省这条规则是不是拍脑袋定的。
- 教训：**机器闸的价值不在"绿了就万事大吉"，而在每次肉眼发现它漏掉的东西时，立刻把那类缺陷补成规则。**

### 新增铁律
- **拒绝抽象，一律落到具体业务动作**（正文/图内标签/表格一体适用）：抽象词黑名单——要素/交付/输入/输出/职责/赋能/协同/资源整合。
- **不许为图面整洁合并节点**：同层主体有实质差异（能开票报价高 vs 没票报价低）必须拆开画，差异就是题眼。
- **术语两级处理，一个都不能漏**：正文首次出现就地解释 + 附录词条，图里的黑话同样算数。
- 章头 lede 固定「本章回答：……？」且逐一对应本章小节。

### 输入形态：兼容零素材（模式 B）
- 新增**第 0 步分流**：访谈素材 → 模式 A（原流程）；一句话 / 行业名 / 零散笔记 → **模式 B 公开资料驱动**（[references/desk-research.md](references/desk-research.md)）；**公司名** → 先联网查清主营与所处环节，把候选行业与主角视角摆给用户选（公司 ≠ 行业，主角是行业里的一类角色）。
- **模式 B 红线**：没访谈时模型会"用行业常理把空缺填满"，编出来读着专业却不报错——最危险的失败模式。故：能写的正常写，**只有访谈才拿得到的（真实利润账、灰色链路、黑话、一线痛点）宁可留白也不推演**，留白用 `warnbox` 显性标出；骨架 5 图在模式 B 下不必备，画不出真东西就不画。
- 模式 B **额外产出访谈提纲**：把每处留白反推成"该去问什么、该问谁"——它把模式 B 变成模式 A。
- description 覆盖新触发语（"分析一下 XX 行业""研究下 XX 这家公司所在的行业""帮我摸个底"）。

### 联网检索（放开 + 定界）
- 触发条件从"1.2/1.3 两节"改为**"这句话凭什么可信"**：客观事实、素材没覆盖但结构需要的内容、受访者的全行业断言（需交叉验证），都先查再写。
- **查到什么程度就停**：每条事实查到能标年份来源即停、关键数字两个独立来源对上就够、查不到就写"无公开统计"。个案经验/公司内部做法/灰色环节不必查，如实写标口径即可。

### 交付
- **交付前必做「重建行业术语」**（第 9 步）：报告改多轮后术语必走样，导 PDF / 同步飞书前必须过一遍，不能省、不能等用户提醒。
- **PDF 与飞书文档平级**：都放最后、都要用户明确同意，问清"要哪个"由用户选，不默认替他决定。
- 新增可选交付通道 [references/lark-sync.md](references/lark-sync.md)（HTML→飞书 XML、SVG→2x PNG、占位符搬运，含 `media-insert` 日志混入 stdout、`block_move_after` 假成功等坑）。模板内置 `@media print { break-inside: avoid }`。

### 整体梳理（去矛盾、去重复）
- 分工原则：**通则只写在 SKILL.md，framework.md 只写"这一节特有的东西"**，避免同一条规则两处各写、改一处漏一处。
- 收敛重复：「零素材怎么办」原写三遍 → 第 0 步给红线、细节归 desk-research.md；「联网查到什么程度」原 SKILL 与 framework 各写一遍 → framework 只留取数特例；铁律「术语两级」+「图里黑话能查到」合并为一条。
- 自检清单从 20 项收敛，删历史包袱（"不再输出 md"等）。

### README 与发布
- 首屏钩子换成真正的差异点：**"别人拿全网二手信息写给行家看，它拿你的一手访谈写给小白看"**（生态搜索显示"访谈转写 → 行业研报"这条路 GitHub 无人认领，同类清一色公开网搜 + PE 腔）。
- Showcase 用**质量闸的真实输出**（从真报告抓出的 12 条 FAIL），零敏感可传播；新增差异化对比表、安全边界、`npx skills add` 一行安装、skills.sh 徽章、LICENSE(MIT)。
- **不收录样例报告**：真实报告含客户访谈的业务细节（灰色环节、利润结构），属敏感内容，不进公开仓库。

## v0.1.3 — 2026-07-14

**商业模式（重点强化）**
- 明确第二章为全文重点，2.2/2.3 不能只用一张概览图带过；能用图讲清就不用大段文字。
- 2.2：行业存在两种以上实质不同的产品/订单/经营模式时，应做对比（优先图形化）；**对比维度不套固定清单**，从素材选 3–6 个最能解释"为何不能用同一套经营逻辑"的变量，图后必须给经营结论——防止千篇一律的固定维度对比表。
- 2.3：目标定为"通过产业链与资金流动讲清怎么挣钱"，**结构不定死**，但必须回答"谁付钱、收入成本在哪、利润来自哪"；交易复杂（多收入源/多笔成本/事后报账/凭证核销/需对账）时增加资金结算闭环图，简单则不硬拆。
- **图数量由认知复杂度决定**（改掉"自由发挥、不设上限"的措辞）：一图一论点、讲不清才拆；详细不等于编，边界是素材撑得住 + 可核实常识。

**图号**
- 四类骨架图改为"固定在章节位置、图号不写死为 1–4、全文按出现顺序连续编号"，修正与"商业模式按需加图"的冲突；自检项同步。

**稳健性（校准）**
- 公开数据从"必须"放软为"有则必用、查不到就明说不硬凑"，新增"大盘衬托主角、不喧宾夺主"约束。
- PDF 导出新增分页检查：长图/宽表跨页断裂的处理（`break-inside: avoid`）。
- 自检新增"过拟合回归"项：换一个结构不同的行业想一遍，规则是否会逼出僵硬套路、是否只对本案例成立。
