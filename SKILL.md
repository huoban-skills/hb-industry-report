# 行业分析报告生成

把一手访谈素材（或公开资料）转化为让行业小白"看得懂、记得住"的行业分析报告。**只讲行业，不讲产品方案。** 一手访谈是主场景；素材少甚至只给行业名/公司名也能做——只写公开资料撑得住的部分，其余留白。

## 产出物

以行业名新建文件夹（默认在素材同目录）：

```
<行业名>/
├── <行业名>行业分析报告.html    # 主交付物，SVG 内嵌，单文件可直接分享
├── <行业名>行业分析报告.pdf     # 可选交付形态（另一种：飞书文档），见第 7 步
└── figures/                     # 每张图的独立 SVG，由 finalize.py 自动导出（配色写死+xmlns），不手工维护
    ├── 01-发展时间轴.svg
    ├── 02-产品服务全景.svg
    ├── 03-业务协作链.svg
    ├── 04-资金流转.svg
    ├── 05-L1业务流程.svg
    └── 06-xxx.svg            # 第 6 张起按内容命名，数量不限
```

HTML 固定骨架：

```
封面（眉题"行业分析报告" + 行业名主标题 + 破折号副题）
摘要（两段，供口头介绍照着念）
目录
一、行业背景     1.1 行业定义 /〔1.2 细分行业〕/ 发展阶段 / 业务规模　（〔细分行业〕仅在主角是"大行业下的一类"时插入，插入则后两节顺延编号；详见 framework）
二、商业模式     2.1 目标客户与需求 / 2.2 产品与服务 / 2.3 上下游业务协作链 / 2.4 资金流转与利益分配
三、需求分析     3.1 L1 总流程 / 3.2 分块详解（4–6 个业务环节）/ 3.3 方案建议
附录、行业术语表（分组）
页脚（资料来源与口径说明）
```

**骨架图 5 类必备**：发展时间轴、产品/服务全景、业务协作链、资金流转、L1 流程——固定在对应章节，图号不写死、全文按出现顺序连续编号。**2.3（谁和谁协作、交接什么，不出现金额）与 2.4（钱和票怎么流、怎么挣钱）必须分立、各配一张图。** 逐节写法见 [references/framework.md](references/framework.md)。

## 图与文字的分工

- **该用图**：关系、结构、流动、顺序、对比、配比——这类内容用文字写，读者得在脑子里自己重建一张图。
- **该用文字**：定义、判断、因果、故事、行业的"人味"——硬塞进图里就退化成一堆抽象名词框。
- **图的多少由认知复杂度决定**：一图一论点，讲不清才拆；骨架之外全文任何位置都可加图。
- **文字不是图的附属品**：全文目标是把陌生行业**生动地讲给小白听**——具体的人、动作、场景，受访者原话、黑话、反常识之处；不把正文写成图的干巴巴注脚。

## 工作流程（素材不足时的逐节差异见 desk-research.md）

**所有输入同一套骨架与流程；深度由证据决定，逐节、逐句生效**——判据是"有没有具体可溯源的证据"，不是"证据是不是访谈"：素材撑得住的写透，没有的查公开资料（第 3 步）；行业内部机制（利润账、灰色链路、黑话、一线痛点）通常只有访谈才拿得到，但**高质量公开证据（司法判决、监管处罚、招股书、深度报道里的从业者原话）查到了同样能撑住，按一手素材对待、标明来源**；什么都撑不住时**宁可留白也不推演**（铁律「素材忠实度」）。没访谈时模型默认用行业常理脑补，编出来的读着专业且不报错——这是本 skill 最危险的失败模式。

0. **开工前检查**：
   - **输入是公司名**：公司 ≠ 行业（主角是行业里的一类角色，写成公司研究就跑题成投研报告）。先联网查清主营与产业链环节，**把候选行业与主角视角摆给用户选**（选谁视角就全变），确认再开工；不默认拿"营收最大的那块"当行业。
   - **素材单薄**（一句话/公司名/零散笔记）：先挑明深度弱一档、征得同意；扩口径、逐节留白、访谈提纲见 [references/desk-research.md](references/desk-research.md)。

1. **通读素材**（大文件分页读完，别只读开头），提炼：行业本质与角色、业务/生产流程、产品与成本结构、黑话、关键数字（带出处）、痛点、敏感/灰色环节。

2. **定行业名与边界**：纯行业称谓（"城投定融行业"），不带"管理/系统/平台/服务方案"等业务词；拿不准先问，其余直接出稿——用户习惯"先看初稿再迭代"。

3. **联网补公开资料**。总原则：**公开资料是佐证，不是第二条叙事线**——正文主线沿素材的真实业务展开（具体的人、动作、订单、原话），公开资料只在证实/校准/冲突处以一两句话出现，不为"显得客观"另铺一套行业通识叙事。
   - **必查三类**（触发条件是"这句话凭什么可信"，不是章节位置）：①可核实的客观事实（数字/年份/政策/行业通行规则/主要玩家）；②素材没覆盖但报告结构需要的内容；③受访者的全行业断言——交叉验证不了就降级成"受访者认为"。
   - **先框范围、一次查完**：先把会落进正文的条目列成清单（stat 卡数字 + 政策年份）再查，别边写边零散查；子任务只回「事实 + 出处 URL」一行一条，别回传整页全文。控的是精读与回传，**不是检索广度**——叫法/口径存疑先按 desk-research.md 扩口径。
   - **不必查**：个案经验、公司内部做法、灰色环节真实操作——如实写、标口径。
   - **查到哪停**：能标年份与来源即停；大盘数字两个独立来源对上就够，软事实不交叉验证；查不到写"无公开统计"，不为填满而编。优先官方来源。**每个数追问"有没有更新的一版"**，取最新年度核对原文（时间轴历史年份除外）。本节取数细则见 framework.md。

4. **读 [references/framework.md](references/framework.md)（写前必读）**，复制 [assets/template.html](assets/template.html) 到 `<行业名>/` 文件夹，逐节填充。CSS 与组件类不改、只填内容；SVG 颜色一律用 `var(--accent)` 等变量。

5. **终稿收口**：`python3 scripts/finalize.py <报告.html>`——重导 `figures/`（改过任何一张内嵌 SVG 都必须重跑，否则独立 SVG 停在旧版）+ 内容与图形机检。**FAIL 清零才能交付；WARN 如实报给用户，改不改由人决定。** 规则变更后须用它回归全部历史报告。

6. **人工自检**：只做机器判不了的（素材忠实度、生动程度、数字溯源、摘要 20 秒能懂），交付时列出需用户核实的点（推断年份、示意数字、可能转述走样的机制）。**图不目检**：硬伤脚本已兜底，不开浏览器/localhost、不读 PNG；好不好看交用户审稿看。

7. **交付（先跟用户确认）**：先重跑第 5 步过闸——报告改多轮后术语必走样（新黑话没进附录、孤儿词条、死链），脚本会抓，修完再交付。然后问用户："内容定稿了吗？要 **PDF**、**飞书文档**，还是都要？"两者平级、都放最后、都要用户明确同意，不默认替他决定。
   - **PDF**：首选无头 Chrome（对内嵌 SVG 还原最忠实）：`"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless --disable-gpu --no-pdf-header-footer --print-to-pdf="<行业名>行业分析报告.pdf" "<行业名>行业分析报告.html"`；无 Chrome 退 weasyprint。**导完必检分页**：长图宽表断裂就微调元素高度重导（模板已内置 `break-inside: avoid`）。
   - **飞书文档**：按 [references/lark-sync.md](references/lark-sync.md)：HTML → 飞书 XML 整篇写入 → [`scripts/export_png.py`](scripts/export_png.py) 把 SVG 导成**固定 3200px 宽** PNG → 逐张插入搬运。别手搓——media-insert 进度日志会混进 stdout，解析失败别重试，图其实已插入。

## 铁律（历次用户纠偏沉淀，违反即返工）

- **纯业务视角**：把"系统模块名"翻译回"业务环节名"（档案管理→债权确权），删掉一切系统操作（按钮/表/自动化/字段/导 Excel）；上下游角色表里不出现软件/系统服务商。
- **标题规范**：章节标题用正式名词（"行业背景"），口语化表述（"钱是怎么欠下的"）放章头 lede；业务块标题纯名词、不带破折号后缀。
- **数字纪律：事实必须真实，推断必须有据，示例可以假设——三者绝不混淆**。每个数字答得出"这数哪来的"，答不出且非示例就删。个案数据标"供建立体感，非行业统计口径"；推断年份图注标"推断"；公开政策背景标"依据公开资料整理，正式引用前请核对原文"。**示例（如资金图的"1 万元订单账"）是把生意讲活的合法手段，不要为客观而删**——但须算术自洽、紧邻标注"示意，非企业真实数据、非行业统计"，只用于解释机制，不得证明行业价格/利润率/规模。
- **敏感内容只描述、不指导**：灰色环节（两套账、无票采购等）如实描述 + `warnbox` 合规提示，绝不给操作指引。
- **术语两级处理，一个不漏**：黑话正文首次出现就地解释（`.term`）+ 附录术语表分组回查，两头都做；**图里的黑话同样算数**。
- **素材忠实度：有的不许丢，补的必须真**。受访者反复强调、占篇幅大的内容大概率是行业要害，漏掉比写错更糟；素材外补充必须可核实并标来源，纯推演要么删、要么显式标"推断"，不许包装成事实填满章节（反例：访谈没提"复盘"，却编出带子流程和痛点的"复盘沉淀"环节）。
- **拒绝抽象，一律落到具体业务动作**（正文/图内标签/表格一体适用）：解释任何角色、环节、概念都要有地名、动作、场景、人。**黑名单**：要素、交付、输入、输出、职责、赋能、协同、资源整合——写这些等于没写。
  ✅「在北京、上海收客的组团社，把'去青海的人'交给当地」「调度车、订酒店、派导游」「报备锁单」「垫资集采」「点A发B」
  ❌「掌握客源，转交履约需求」「谁把要素喂给主角」「主角向谁交付」
- **不为图面整洁合并节点**：同层主体有实质差异（品牌商能开票报价高 vs 散户没票报价低）必须拆开画——差异就是行业的题眼。
