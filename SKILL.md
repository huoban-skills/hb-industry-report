# 行业分析报告生成器

把一手访谈素材转化为让行业小白"看得懂、记得住"的行业分析报告。读者是即将进入该行业做竞品调研和产品设计的产品经理，但报告本身**只讲行业，不讲产品方案**。

## 产出物

单文件 HTML，保存为 `<行业名>行业分析报告.html`（默认放在素材同目录）。固定骨架：

```
封面（眉题"行业分析报告" + 行业名主标题 + 破折号副题）
摘要（三段，供口头介绍照着念）
目录
一、行业背景     1.1 行业定义 / 1.2 发展阶段 / 1.3 业务规模
二、商业模式     2.1 目标客户与需求 / 2.2 产品与服务 / 2.3 资金上下游与利益分配
三、需求分析     3.1 L1 总流程 / 3.2 分块详解（4–6 个业务环节）/ 3.3 方案切入视角
附录、行业术语表（分组）
页脚（资料来源与口径说明）
```

标配 5 张内嵌 SVG 图：发展时间轴、产品/服务全景、资金链双层图、L1 流程图、行业特色图（挑最能代表该行业复杂性的机制画一张，如多段计息、利润漏斗）。

## 工作流程

1. **通读素材**。大文件分页读完，不要只读开头。边读边提炼六类事实：行业本质与角色、业务流程、行业黑话、关键数字（带出处）、痛点、敏感/灰色环节。
2. **定行业名与边界**。行业名必须是纯行业称谓（"城投定融行业"），不带"管理/系统/平台/服务方案"这类业务词。行业名或边界拿不准时先问用户，其余情况直接出稿——用户习惯"先看初稿再迭代"。
3. **读 [references/framework.md](references/framework.md)**（写前必读），按逐节指南组织内容。
4. **复制 [assets/template.html](assets/template.html)** 为目标文件，逐节填充。CSS 与组件类不要改，只填内容；SVG 颜色一律用 `var(--accent)` 等 CSS 变量。
5. **自检**（见下方清单），交付时列出需用户核实的点（推断的年份、示意数字、转述可能走样的机制）。

## 铁律（历次用户纠偏沉淀，违反任何一条都要返工）

- **纯业务视角**。访谈常常是围着某个系统演示聊的，但报告只写真实业务流程——把"系统模块名"翻译回"业务环节名"（档案管理→债权确权），删掉一切系统操作（按钮、表、自动化、导 Excel、字段）；上下游角色表里不出现软件/系统服务商。
- **标题规范**。章节标题用正式名词（"行业背景"），口语化表述（"钱是怎么欠下的"）放进章头 lede；业务块标题是纯名词，不带破折号后缀。
- **数字纪律**。写完自问每个数字"这数哪来的"——答不出就删。单一案例数据标注"供建立体感，非行业统计口径"；示意测算标"示意"；推断年份在图注标"推断"；公开资料补充的政策背景标"依据公开资料整理，正式引用前请核对原文"。
- **敏感内容只描述、不指导**。灰色业务环节（两套账、无票采购等）如实描述现状，配 `warnbox` 合规风险提示，绝不给操作指引。
- **术语两级处理**。黑话在正文首次出现处就地解释（用 `.term` 样式），附录术语表分组集中供回查——不要只做其中一头。

## 自检清单

- [ ] 全文 grep 不到"系统/软件/按钮/自动化/字段/导入导出"等系统维度用语（行业固有称谓如"投融资平台"除外）
- [ ] 摘要三段齐全：是什么+所处阶段 / 生意怎么转+环节链 / 痛点根因+一句话核心诉求
- [ ] 每个业务块五段齐全：做什么 / 子流程 / 谁在参与 / 业务特征 / 痛点
- [ ] 图 1–5 编号连续，figcaption 完整，SVG 无写死的颜色值
- [ ] 每个数字可溯源，口径标注齐全；页脚有资料来源
- [ ] 封面无"读者/目的/版本"行
