# 行业分析报告生成

把一手访谈素材转化为让行业小白"看得懂、记得住"的行业分析报告。读者是即将进入该行业做竞品调研和产品设计的产品经理，但报告本身**只讲行业，不讲产品方案**。

## 产出物

以行业名新建一个文件夹（默认在素材同目录），三件套：

```
<行业名>/
├── <行业名>行业分析报告.html    # 主交付物，SVG 内嵌，单文件可直接分享
├── <行业名>行业分析报告.md      # 报告正文 markdown 版，图以相对路径引用 figures/
└── figures/                     # 每张图的独立 SVG，便于复用到飞书/PPT
    ├── 01-发展时间轴.svg
    ├── 02-产品服务全景.svg
    ├── 03-上下游链条全景.svg
    ├── 04-L1业务流程.svg
    └── 05-xxx.svg            # 第 5 张起按内容命名，数量不限
```

注意：独立 SVG 脱离页面后 `var()` 不生效，必须在每个 .svg 文件内嵌 `<style>` 把配色变量按浅色主题值写死。

HTML 报告的固定骨架：

```
封面（眉题"行业分析报告" + 行业名主标题 + 破折号副题）
摘要（三段，供口头介绍照着念）
目录
一、行业背景     1.1 行业定义 / 1.2 发展阶段 / 1.3 业务规模
二、商业模式     2.1 目标客户与需求 / 2.2 产品与服务 / 2.3 上下游与利益分配（资金链/产业链，用词随行业）
三、需求分析     3.1 L1 总流程 / 3.2 分块详解（4–6 个业务环节）/ 3.3 方案切入视角
附录、行业术语表（分组）
页脚（资料来源与口径说明）
```

**骨架图 4 张必备**：发展时间轴、产品/服务全景、上下游链条全景图（资金链或产业链）、L1 流程图——这四张每份报告同构，保证读者跨报告的阅读习惯。**其余的图自由发挥、不设上限**：能用图讲清楚的就不要用大段文字（行业特色机制、利润漏斗、成本瀑布、多段计息这类复杂机制都值得成图）；文字只承担两件事——重点结论，和必要的场景化描述。图编号按出现顺序连续。

## 工作流程

1. **通读素材**。大文件分页读完，不要只读开头。边读边提炼这些事实：行业本质与角色、业务/生产流程、产品与成本结构（实体产品业尤其重要）、行业黑话、关键数字（带出处）、痛点、敏感/灰色环节。
2. **定行业名与边界**。行业名必须是纯行业称谓（"城投定融行业"），不带"管理/系统/平台/服务方案"这类业务词。行业名或边界拿不准时先问用户，其余情况直接出稿——用户习惯"先看初稿再迭代"。
3. **读 [references/framework.md](references/framework.md)**（写前必读），按逐节指南组织内容。
4. **建 `<行业名>/` 文件夹，复制 [assets/template.html](assets/template.html)** 为其中的 HTML 文件，逐节填充。CSS 与组件类不要改，只填内容；SVG 颜色一律用 `var(--accent)` 等 CSS 变量。
5. **派生 markdown 与独立图**：把每张 SVG 另存到 `figures/`（文件内嵌 `<style>` 写死浅色配色）；从 HTML 提取正文生成 `.md`（章节结构一致，表格转 markdown 表，图改为引用 `figures/` 相对路径）。
6. **图形自检（零 token）**：跑 `python3 scripts/check_svg_overlap.py <报告.html>`，它用几何计算检查"线穿文字"和"文字超出画布"——**FAIL 必须清零**，WARN（<5px 贴边）人工判断即可。不要用浏览器截图逐张肉眼检查（费 token），截图只在启用全新图型时抽查一次。
7. **内容自检**（见下方清单），交付时列出需用户核实的点（推断的年份、示意数字、转述可能走样的机制）。

## 铁律（历次用户纠偏沉淀，违反任何一条都要返工）

- **纯业务视角**。访谈常常是围着某个系统演示聊的，但报告只写真实业务流程——把"系统模块名"翻译回"业务环节名"（档案管理→债权确权），删掉一切系统操作（按钮、表、自动化、导 Excel、字段）；上下游角色表里不出现软件/系统服务商。
- **标题规范**。章节标题用正式名词（"行业背景"），口语化表述（"钱是怎么欠下的"）放进章头 lede；业务块标题是纯名词，不带破折号后缀。
- **数字纪律**。写完自问每个数字"这数哪来的"——答不出就删。单一案例数据标注"供建立体感，非行业统计口径"；示意测算标"示意"；推断年份在图注标"推断"；公开资料补充的政策背景标"依据公开资料整理，正式引用前请核对原文"。
- **敏感内容只描述、不指导**。灰色业务环节（两套账、无票采购等）如实描述现状，配 `warnbox` 合规风险提示，绝不给操作指引。
- **术语两级处理**。黑话在正文首次出现处就地解释（用 `.term` 样式），附录术语表分组集中供回查——不要只做其中一头。
- **素材忠实度：有的不许丢，补的必须真**。素材里的关键事实一个不能丢——受访者反复强调、访谈里占篇幅大的内容，大概率就是行业要害（如"每个团必须向监管平台上报""市面软件都为批发商设计"），漏掉它们比写错更糟。素材之外允许补充，但必须是客观存在、可核实的事实（公开统计、政策文件、公认行业常识），标注来源；纯推演要么删掉、要么显式标"推断"，不许包装成事实去填满章节（反例：访谈没提"复盘"，却编出一个带子流程和痛点的"复盘沉淀"环节）。
- **拒绝岗位说明书式的抽象描述，全文适用**。解释任何角色、环节、概念，都要落到具体例子——地名、动作、场景、人："在北京、上海收客的组团社，把'去青海的人'交给当地""调度车、订酒店、派导游"，而不是"掌握客源，转交履约需求"这类职责套话。图内副标题、正文、表格单元格一体适用；利益分配这类机制，优先用"顺着一笔钱走一遍"的叙事讲。

## 自检清单

- [ ] 全文 grep 不到"系统/软件/按钮/自动化/字段/导入导出"等系统维度用语（行业固有称谓如"投融资平台"除外）
- [ ] 摘要两段、三四句话：20 秒能读懂"是什么行业、难在哪、要什么"
- [ ] 每个业务块五段齐全：做什么 / 子流程 / 谁在参与 / 业务特征 / 痛点
- [ ] 骨架图 1–4 齐全，全部图编号按出现顺序连续，figcaption 完整，SVG 无写死的颜色值
- [ ] `check_svg_overlap.py` 通过：无线穿文字、无文字出画布（FAIL=0）
- [ ] 每个数字可溯源，口径标注齐全；页脚有资料来源
- [ ] 对照素材回读一遍：受访者强调的事实全部进了报告；素材外补充的内容均可核实或已标"推断"
- [ ] 封面无"读者/目的/版本"行
- [ ] 文件夹三件套齐全：HTML、md、figures/；md 里的图片路径能正常显示，独立 SVG 自带配色
