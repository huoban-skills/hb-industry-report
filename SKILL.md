# 行业分析报告生成

把一手访谈素材（或公开资料）转化为让行业小白"看得懂、记得住"的行业分析报告。报告本身**只讲行业，不讲产品方案**。

一手访谈是主场景——它能讲透行业内部的真实机制；零素材（只给行业名或公司名）也能做，但只写公开资料撑得住的部分，其余留白（见第 0 步）。

## 产出物

以行业名新建一个文件夹（默认在素材同目录）：

```
<行业名>/
├── <行业名>行业分析报告.html    # 主交付物，SVG 内嵌，单文件可直接分享
├── <行业名>行业分析报告.pdf     # 可选交付形态之一，由 HTML 导出
└── figures/                     # 每张图的独立 SVG，便于复用到飞书/PPT
    ├── 01-发展时间轴.svg
    ├── 02-产品服务全景.svg
    ├── 03-业务协作链.svg
    ├── 04-资金流转.svg
    ├── 05-L1业务流程.svg
    └── 06-xxx.svg            # 第 6 张起按内容命名，数量不限
```

注意：独立 SVG 脱离页面后 `var()` 不生效，必须在每个 .svg 文件内嵌 `<style>` 把配色变量按浅色主题值写死。

HTML 是编辑与迭代的载体；**PDF 与飞书文档是两种可选的最终交付形态，平级**——都放在最后一步，都要先跟用户确认（内容定稿没有、要哪一种、还是都要），由用户自己选，别默认替他决定。

HTML 报告的固定骨架：

```
封面（眉题"行业分析报告" + 行业名主标题 + 破折号副题）
摘要（两段，供口头介绍照着念）
目录
一、行业背景     1.1 行业定义 / 1.2 发展阶段 / 1.3 业务规模
二、商业模式     2.1 目标客户与需求 / 2.2 产品与服务 / 2.3 上下游业务协作链 / 2.4 资金流转与利益分配
三、需求分析     3.1 L1 总流程 / 3.2 分块详解（4–6 个业务环节）/ 3.3 方案建议
附录、行业术语表（分组）
页脚（资料来源与口径说明）
```

**骨架图 5 类必备**：发展时间轴、产品/服务全景、上下游业务协作链图、资金流转图、L1 流程图——每份报告同构，固定在对应章节、图号不写死，全文按出现顺序连续编号（任何位置加图都会把后面的图号顺延）。其中 **2.3（谁和谁协作、交接什么，不出现金额）与 2.4（钱和票怎么流、怎么挣钱）必须分立成两节、各配一张图**，一张图塞两件事读者必然看不懂——逐节写法见 [references/framework.md](references/framework.md)。

## 图与文字的分工

**骨架之外要不要加图，全文任何位置都由你判断，不限于某一章**。判断的是内容形态，不是章节位置：

- **该用图**——关系、结构、流动、顺序、对比、配比（谁和谁怎么咬合、钱怎么走、成本怎么分、几种模式有何不同、一件事分几步）。这类内容用文字写，读者要在脑子里重建一张图，费力且容易走样。
- **该用文字**——定义、判断、因果、故事、行业的"人味"（为什么会变成这样、这行的人怎么想、一个具体场景里发生了什么）。这类内容硬塞进图里，就会退化成一堆抽象名词框。

**图的多少由认知复杂度决定，不是越多越好**：一张图只讲一个论点，讲不清才拆图；详细的边界是素材撑得住 + 可核实常识，不为凑详细而编。

**文字不是图的附属品。** 全文的目标是把一个陌生行业**生动形象地讲给小白听**，所以文字本身就得生动、具体、场景化——讲清结论之外，更要把行业讲活：具体的人、具体的动作、具体的场景，受访者的原话、行业里的黑话和那些反常识的地方。不要因为"能用图讲清就不用文字"，就把正文写成图的干巴巴的注脚。

## 第 0 步：看输入是什么，决定走哪条路

**动笔前先分流。两种模式共用同一套章节骨架，区别只在"哪些内容有资格写"。**

| 输入                                                     | 模式                 | 怎么走                                                       |
| -------------------------------------------------------- | -------------------- | ----------------------------------------------------------- |
| 访谈记录 / 调研纪要 / 录音转写                           | **A · 一手素材驱动** | 正常走下面的工作流程                                        |
| 一句话（"分析一下 XX 行业"）、几条零散笔记、**一家公司名** | **B · 公开资料驱动** | 见 [references/desk-research.md](references/desk-research.md)；公司名先按下方前置翻译成行业 |

**公司名前置（走 B 前先做）**：公司 ≠ 行业，而且本 skill 的主角是行业里的**一类角色**，不是某家公司（写成公司研究就跑题成投研报告了）。先联网查清这家公司主营什么、处在哪条产业链的哪个环节，然后**把候选的行业与主角视角摆给用户选**（一家公司常横跨多个环节，选谁整份报告的视角就全变了），确认后再按 B 开写。不要自己默认拿"营收最大的那块"当行业。

**模式 B 的红线**：没有访谈时，模型的默认行为是**用行业常理把空缺填满**——编出来的东西读着专业、实则全是脑补，且不会报错，这是本 skill 最危险的失败模式。铁律：**只有访谈才拿得到的（真实利润账、灰色链路、行业黑话、一线痛点）宁可留白也不推演**。开工前先跟用户挑明深度会弱一档、征得同意；逐节哪些能写哪些必须留白、留白怎么标、访谈提纲怎么出，全在 [references/desk-research.md](references/desk-research.md)。

## 工作流程（模式 A；模式 B 的差异见 desk-research.md）

1. **通读素材**。大文件分页读完，不要只读开头。边读边提炼这些事实：行业本质与角色、业务/生产流程、产品与成本结构（实体产品业尤其重要）、行业黑话、关键数字（带出处）、痛点、敏感/灰色环节。

2. **定行业名与边界**。行业名必须是纯行业称谓（"城投定融行业"），不带"管理/系统/平台/服务方案"这类业务词。行业名或边界拿不准时先问用户，其余情况直接出稿——用户习惯"先看初稿再迭代"。

3. **联网补公开资料**。先立一条总原则——**公开资料是佐证，不是第二条叙事线**：模式 A 下正文主线始终沿素材里的真实业务展开（具体的人、动作、订单、原话），公开资料只在能证实、校准或揭示冲突的位置以一两句话出现。不要为了"显得客观"另铺一套行业通识叙事把个案故事冲淡——那样的报告高不成低不就，读者记不住核心。

   **触发条件不是章节位置，而是"这句话凭什么可信"**——全文任何位置，只要要写的东西属于下面三类，就必须先查再写（1.2/1.3 只是最密集的地方，不是唯一）：

   - **可核实的客观事实**：数字、年份、政策法规、监管要求、行业通行规则、公开的主要玩家与产业格局；
   - **素材没覆盖、但报告结构需要的内容**（素材稀薄时这部分会很大）；
   - **受访者的全行业断言**（"这行都这么干""市面软件都是给批发商做的"）——素材只能证明"他这么认为"，写成行业事实前要交叉验证，验证不了就降级成"受访者认为"。

   **先框范围、一次查完（控 token）**：动手前把会真正落进正文的条目列成清单（stat 卡大盘数字 + 要标注的政策年份），一次查完，别边写边零散查；派联网子任务只回「事实 + 出处 URL」一行一条，**别回传整页全文、别把二三十个网页拖进上下文**。

   **不必查的**：受访者个案经验、公司内部做法、灰色环节真实操作——如实写、标清口径即可，别为"有出处"去网上找二手说法附会。

   **查到哪停**：每条查到**能标出年份与来源**即停；关键大盘数字（规模、利润率、渗透率）**两个独立来源对上**就够，别凑第三个，软事实不交叉验证；查不到就写"无公开统计"，靠个案 + 可核实常识撑，**绝不为填满而编**。优先官方来源（部委/协会/统计公报）。**每个数查完追问一句"有没有更新的一版"**——蓝皮书、统计公报、市场规模都是年度更新的，别拿搜索里第一条旧的就用；取最新年度核对原文（时间轴的历史年份除外）。本节特有取数规则见 framework.md。

4. **读 [references/framework.md](references/framework.md)**（写前必读），按逐节指南组织内容。

5. **建 `<行业名>/` 文件夹，复制 [assets/template.html](assets/template.html)** 为其中的 HTML 文件，逐节填充。CSS 与组件类不要改，只填内容；SVG 颜色一律用 `var(--accent)` 等 CSS 变量。

6. **派生独立图**：`figures/` 是 HTML 内嵌图的派生物，不独立维护。报告编写阶段不必手工同步；第 7 步的 [`scripts/finalize.py`](scripts/finalize.py) 会整套重导，写死浅色配色并补 `xmlns`，保证脱离 HTML 后能单独打开。⚠️ 只要改过任何一张内嵌 SVG（哪怕只挪个标签），交付前都必须重新跑 `finalize.py`，否则独立 SVG 会停在旧版、和报告对不上。

7. **终稿收口（FAIL 必须清零）**

   ```bash
   python3 scripts/finalize.py <报告.html>
   ```

   脚本会重导 `figures/`，并完成内容与图形检查。内嵌 SVG 引用未声明的 CSS 变量、或独立 SVG 导出后仍残留 `var()`，都按 **FAIL** 处理。**FAIL 清零后才能交付，WARN 由人工判断。** Skill 规则变更后，须用该命令回归全部历史报告。

8. **人工自检**（见下方清单里机器判不了的部分：素材忠实度、生动程度、数字溯源、摘要是否 20 秒能懂），交付时列出需用户核实的点（推断的年份、示意数字、转述可能走样的机制）。

   **图不用你目检（省 token）**：图形硬伤（穿字/裁切/压色块/压字）第 7 步的 `check_svg_overlap.py` 已经兜底——**不要开浏览器 / localhost、也不要读 PNG 去肉眼看图**，那是重复劳动还费 token。好不好看、读不读得顺，交给用户审稿时看。脚本报的 `WARN`（宽度按 CJK≈1em 估算，边界会误报）**如实报给用户就行，改不改由人决定**——你不用自己开图去判断。

9. **交付（最后一步，先跟用户确认）**。

   **质量闸：不过闸不许交付。** 导 PDF / 同步飞书**之前**重跑一遍第 7 步的 `finalize.py`，总 **FAIL 必须为 0**。报告改过多轮之后术语一定走样（正文新写的黑话没进附录、删掉的段落留下孤儿词条、锚点变成死链），脚本会替你抓出来——**修完再交付，别把 FAIL 带进 PDF**。术语表是这份报告给小白的"回查入口"，它烂了整份报告的可用性就塌一半。

   闸过了，再**问用户**："内容定稿了吗？要出 **PDF**、同步到**飞书文档**，还是都要？"——**两者都是收尾动作、都放最后、都要用户明确同意才做，由用户自己选**，不要默认帮他决定，也不要没问就开始导。用户若还要改内容，回到 HTML 迭代，改完重新确认再交付（术语同样要再过一遍）。

   - **PDF**：由 HTML 导出同名 `.pdf`——首选无头 Chrome（对内嵌 SVG 还原最忠实）：`"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless --disable-gpu --no-pdf-header-footer --print-to-pdf="<行业名>行业分析报告.pdf" "<行业名>行业分析报告.html"`；若无 Chrome 再退到 weasyprint 等。**导完必检分页**：长图、宽表被切断是常见坑（模板已内置 `@media print` 的 `break-inside: avoid`，仍要抽查一遍），有断裂就微调元素高度后重导。
   - **飞书文档**：用户给出文档链接/token 时，按 [references/lark-sync.md](references/lark-sync.md) 走：HTML → 飞书 XML 整篇写入 → 用 [`scripts/export_png.py`](scripts/export_png.py) 把 SVG 导成**固定 3200px 宽**的 PNG → 逐张插入并搬到占位处。不要再用“2x”这种依赖原始画布尺寸的模糊口径；3200px 是飞书插图的固定清晰度标准。别手搓，流程里有坑（媒体插入的进度日志会混进 stdout，解析失败别重试，图其实已经插进去了）。

## 铁律（历次用户纠偏沉淀，违反任何一条都要返工）

- **纯业务视角**。访谈常常是围着某个系统演示聊的，但报告只写真实业务流程——把"系统模块名"翻译回"业务环节名"（档案管理→债权确权），删掉一切系统操作（按钮、表、自动化、导 Excel、字段）；上下游角色表里不出现软件/系统服务商。
- **标题规范**。章节标题用正式名词（"行业背景"），口语化表述（"钱是怎么欠下的"）放进章头 lede；业务块标题是纯名词，不带破折号后缀。
- **数字纪律：事实必须真实，推断必须有据，示意可以假设——但三者绝不许混淆**。写完自问每个数字"这数哪来的"——答不出且不是示意就删。单一案例数据标注"供建立体感，非行业统计口径"；推断年份在图注标"推断"；公开资料补充的政策背景标"依据公开资料整理，正式引用前请核对原文"。**示例（如资金图里的"1 万元订单账"）允许人为假设——它是把生意讲活的合法手段，不要为了客观而删掉**，但必须算术自洽、紧邻标注"示意，非企业真实数据、非行业统计"，且只用于解释机制（成本怎么分、利润怎么被吃掉），不得用来证明行业价格、平均利润率或市场规模。
- **敏感内容只描述、不指导**。灰色业务环节（两套账、无票采购等）如实描述现状，配 `warnbox` 合规风险提示，绝不给操作指引。
- **术语两级处理，一个都不能漏**。黑话在正文首次出现处就地解释（`.term` 样式）+ 附录术语表分组集中供回查，两头都要做。**图里写了的黑话同样算数**——图内出现"供应链金融""清分""报备"，读者就一定会问这是啥。
- **素材忠实度：有的不许丢，补的必须真**。素材里的关键事实一个不能丢——受访者反复强调、访谈里占篇幅大的内容，大概率就是行业要害（如"每个团必须向监管平台上报""市面软件都为批发商设计"），漏掉它们比写错更糟。素材之外允许补充，但必须是客观存在、可核实的事实（公开统计、政策文件、公认行业常识），标注来源；纯推演要么删掉、要么显式标"推断"，不许包装成事实去填满章节（反例：访谈没提"复盘"，却编出一个带子流程和痛点的"复盘沉淀"环节）。
- **拒绝抽象，一律落到具体业务动作——正文、图内标签、表格单元格一体适用**。解释任何角色、环节、概念，都要有地名、动作、场景、人。**抽象词黑名单**：要素、交付、输入、输出、职责、赋能、协同、资源整合——写这些等于没写，见到即判不合格。
  ✅「在北京、上海收客的组团社，把'去青海的人'交给当地」「调度车、订酒店、派导游」「私下透商机」「报备锁单」「垫资集采」「点A发B」
  ❌「掌握客源，转交履约需求」「谁把要素喂给主角」「主角向谁交付」
- **不许为了图面整洁合并节点**。同一层里若两类主体有**实质差异**（品牌商报价高能开票 vs 本地散户报价低没发票；组团社出客源 vs 车队酒店出运力），必须分开画——那个差异往往就是行业的题眼，合并掉等于把最值钱的信息抹了。

> 🤖 标记的项由 `scripts/finalize.py` 统一触发机检，不要人肉过——人肉必漏。跑脚本，FAIL 清零。
> 其余项机器判不了（素材忠实度、生动程度、是否 20 秒能读懂），必须自己回读。

- [ ] 全文 grep 不到"系统/软件/按钮/自动化/字段/导入导出"等系统维度用语（行业固有称谓如"投融资平台"除外）
